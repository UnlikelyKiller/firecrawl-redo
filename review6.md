**Findings**

- High: `ScrapeWorker` can leave jobs stuck in `RUNNING` if Multilogin eligibility resolution throws. `processJob()` updates the job to `RUNNING`, then awaits `buildEngines()`, but only handles `Result` failures, not exceptions. Any thrown error from `resolveEligibility()` or its dependencies will bypass the later `FAILED` status update and return path. See [worker.ts](/C:/dev/firecrawl-redo/packages/jobs/src/worker.ts:68), [worker.ts](/C:/dev/firecrawl-redo/packages/jobs/src/worker.ts:80), [worker.ts](/C:/dev/firecrawl-redo/packages/jobs/src/worker.ts:164).

- High: the queued worker path does not actually support domain-specific named Multilogin profiles. The API-side resolver can return a per-domain `profileId`, but `MultiloginOptions.resolveEligibility` cannot return one, and the worker always instantiates `MultiloginCdpEngine` from the static constructor options. That means required named-profile domains can only use one globally preconfigured profile or fail as unconfigured on the jobs path. See [multilogin.ts](/C:/dev/firecrawl-redo/apps/api/src/routes/crawlx/v2/multilogin.ts:60), [multilogin.ts](/C:/dev/firecrawl-redo/apps/api/src/routes/crawlx/v2/multilogin.ts:71), [multilogin.ts](/C:/dev/firecrawl-redo/apps/api/src/routes/crawlx/v2/multilogin.ts:122), [worker.ts](/C:/dev/firecrawl-redo/packages/jobs/src/worker.ts:21), [worker.ts](/C:/dev/firecrawl-redo/packages/jobs/src/worker.ts:177).

- Medium: the bridge is effectively unauthenticated when `MULTILOGIN_SHARED_SECRET` is unset. The config makes the secret optional, and `authorizeRequest()` returns `true` for all requests if it is missing, which exposes attach/release/CDP proxy operations to any reachable caller. That is a real control-plane risk if the bridge is ever rebound off loopback, tunneled, or misconfigured. See [config.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/config.ts:48), [config.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/config.ts:125), [auth.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/auth.ts:118), [auth.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/auth.ts:120).

- Medium: the new lease model is not durable even though the rollout adds DB schema for it. The bridge runtime keeps lease ownership only in an in-memory `LeaseManager`; it never persists or reloads `browser_profile_leases`. After a bridge restart, active leases are forgotten, so profile contention and cooldown state are lost even though the schema now implies those guarantees. See [runtime.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/runtime.ts:45), [runtime.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/runtime.ts:62), [runtime.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/runtime.ts:153), [lease-manager.ts](/C:/dev/firecrawl-redo/apps/multilogin-bridge/src/lease-manager.ts:39), [migration-smoke.test.ts](/C:/dev/firecrawl-redo/packages/db/src/__tests__/migration-smoke.test.ts:113).

- Low: the engine never heartbeats an attached lease, and there is no coverage for long-running sessions. Today that is masked by the default 15-minute TTL, but if TTL is reduced or a scrape runs long, the bridge can expire and stop the profile mid-run. Current tests only cover attach/release success paths. See [multilogin-cdp.ts](/C:/dev/firecrawl-redo/packages/waterfall-engine/src/engines/multilogin-cdp.ts:88), [multilogin-cdp.ts](/C:/dev/firecrawl-redo/packages/waterfall-engine/src/engines/multilogin-cdp.ts:159), [multilogin-cdp.test.ts](/C:/dev/firecrawl-redo/packages/waterfall-engine/src/__tests__/multilogin-cdp.test.ts:58).

- Low: migration smoke coverage is still thin for the risk in this rollout. It verifies added columns and FK wiring, but not rollout-critical behaviors such as backfilling existing `browser_profiles` rows, enforcing one active lease per profile, or any cooldown/status invariants on `browser_profile_leases`. See [migration-smoke.test.ts](/C:/dev/firecrawl-redo/packages/db/src/__tests__/migration-smoke.test.ts:28), [migration-smoke.test.ts](/C:/dev/firecrawl-redo/packages/db/src/__tests__/migration-smoke.test.ts:113).

**Open Questions / Assumptions**

- The current API resolver appears to assume a single globally configured `MULTILOGIN_PROFILE_ID`, not true per-domain profile selection from `browser_profiles`. If that is intentional for phase 1, it should be stated explicitly because the schema suggests a broader capability.
- I did not find evidence that the queue-worker path is wired to the same DB-backed policy resolver used by the API routes; if that wiring is still pending, the named-profile gap above becomes a rollout blocker rather than just a limitation.
